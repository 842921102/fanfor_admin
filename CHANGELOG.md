# 更新记录

版本号与根目录 [`VERSION`](./VERSION) 一致。发布新版本时请同步更新：

- `VERSION`
- 本文件（在顶部新增一节）
- `mini-fan-package/package.json` 的 `version`
- `mini-fan-package/src/manifest.json` 的 `versionName`；`versionCode` 建议递增整数（如 `100` → `101`）

建议遵循 [语义化版本](https://semver.org/lang/zh-CN/)：`主版本.次版本.修订号`。

### 日常提交 vs 发版

- **平时改代码**：正常 `commit` / `push` 即可，**不必**每次都改版本号。
- **打算对外算「一个新版本」时**（例如要发小程序审核、给客户看更新说明、管理后台要显示新号）：再按下面清单 **一次性** 改完并提交（建议单独一个 commit，例如 `chore: release 1.0.1`）。

### 发版时改哪些（清单）

1. 根目录 **`VERSION`**：改成新版本号（一行，如 `1.0.1`）。
2. **`CHANGELOG.md`**：在 `## [未发布]` **之上**新增一节 `## [1.0.1] - 年-月-日`，写清本版改动；`[未发布]` 里可清空或保留草稿。
3. **根目录 `package.json`**：`version` 字段与上面一致。
4. **小程序**：`mini-fan-package/package.json` 的 `version`；`mini-fan-package/src/manifest.json` 的 `versionName`（与语义化版本一致）及 **`versionCode`**（整数，每次上架前递增，如 `100` → `101`）。
5. **管理后台**：默认读仓库根目录的 `VERSION`（`admin-backend/../VERSION`）。若线上只部署 `admin-backend` 且没有上级 `VERSION` 文件，在部署环境 `.env` 设置 **`FANFOR_RELEASE_VERSION=1.0.1`** 与发版号一致。

**版本号怎么选（简记）**

- **修订号 +1**（`1.0.0` → `1.0.1`）：修 bug、小调整、无新能力约定。
- **次版本 +1**（`1.0.x` → `1.1.0`）：新功能、兼容旧用法。
- **主版本 +1**（`1.x` → `2.0.0`）：大改、破坏性变更。
